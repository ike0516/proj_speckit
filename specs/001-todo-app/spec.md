# Feature Specification: Simple Todo App

**Feature Branch**: `001-todo-app`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: User description: "シンプルなTodoアプリ - タスクの作成・完了・削除ができるWebアプリケーション"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - タスクの作成 (Priority: P1)

ユーザーは新しいタスクをテキストで入力し、リストに追加できる。タスクは入力欄にタイトルを入力して送信することで即座にリストに表示される。

**Why this priority**: タスクの作成はTodoアプリの根幹機能であり、これなしには他の機能も成立しない。最小限のMVPを構成する。

**Independent Test**: 入力欄にテキストを入力して送信し、タスクがリストに表示されることを確認することで単独テスト可能。リスト表示機能のみでユーザーへの価値を提供できる。

**Acceptance Scenarios**:

1. **Given** タスクリストが表示されている, **When** ユーザーが入力欄にタスク名を入力して送信する, **Then** そのタスクが未完了状態でリストの末尾に追加される
2. **Given** 入力欄が空の状態, **When** ユーザーが送信操作をする, **Then** タスクは追加されず、入力欄にエラーメッセージが表示される
3. **Given** タスクが追加された直後, **When** ページを再読み込みする, **Then** 追加したタスクが引き続き表示される

---

### User Story 2 - タスクの完了 (Priority: P2)

ユーザーは未完了のタスクを完了状態に切り替えることができる。完了済みタスクは視覚的に区別して表示される。また、誤って完了にした場合は未完了に戻せる。

**Why this priority**: 完了状態の管理はTodoアプリの中心的な価値であり、進捗を把握するために不可欠。タスク作成後に単独でテストできる。

**Independent Test**: タスクが1件以上ある状態で完了チェックをトグルし、表示が切り替わることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 未完了タスクがリストに表示されている, **When** ユーザーがそのタスクの完了チェックボックスをオンにする, **Then** タスクが完了状態として視覚的に区別して表示される（例: 打ち消し線）
2. **Given** 完了済みタスクがリストに表示されている, **When** ユーザーが完了チェックボックスをオフにする, **Then** タスクが未完了状態に戻る
3. **Given** タスクの完了状態を変更した直後, **When** ページを再読み込みする, **Then** 変更した完了状態が維持されている

---

### User Story 3 - タスクの削除 (Priority: P3)

ユーザーは不要になったタスクをリストから完全に削除できる。完了済み・未完了に関わらず削除できる。

**Why this priority**: リストを整理する機能として重要だが、作成・完了の後に補完的に使われるため優先度は低い。

**Independent Test**: タスクが1件以上ある状態で削除操作をし、リストから消えることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** タスクがリストに表示されている, **When** ユーザーがそのタスクの削除ボタンを押す, **Then** タスクがリストから即座に消える
2. **Given** 完了済みタスクがリストに表示されている, **When** ユーザーが削除ボタンを押す, **Then** 完了済みタスクもリストから即座に消える
3. **Given** タスクが削除された直後, **When** ページを再読み込みする, **Then** 削除したタスクは表示されない

---

### Edge Cases

- タスク名が空白のみの場合、送信を受け付けない
- 非常に長いタスク名（例: 200文字以上）は適切に折り返しまたは切り詰めて表示する
- タスクが0件の場合、リストが空であることを示すメッセージを表示する
- 全タスクを削除した後でも、新規タスクを追加できる

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはユーザーがテキストを入力して新しいタスクを作成できなければならない
- **FR-002**: システムはすべてのタスクを一覧形式で表示しなければならない
- **FR-003**: ユーザーは各タスクの完了・未完了状態をトグルで切り替えられなければならない
- **FR-004**: システムは完了済みタスクと未完了タスクを視覚的に区別して表示しなければならない
- **FR-005**: ユーザーは任意のタスクをリストから削除できなければならない
- **FR-006**: システムはタスクのデータをページ再読み込み後も維持しなければならない（永続化）
- **FR-007**: システムは空のタスク名の送信を拒否し、ユーザーに入力を促さなければならない

### Key Entities

- **タスク（Task）**: ユーザーが管理する作業単位。タスク名（テキスト）、完了状態（完了/未完了）、作成日時を持つ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは30秒以内に新しいタスクを作成してリストに表示できる
- **SC-002**: タスクの完了・削除操作はページを再読み込みせずに即座に画面に反映される
- **SC-003**: すべてのタスクが1画面内でスクロールせずに確認できる（タスク数が少ない通常利用時）
- **SC-004**: 初めて使うユーザーの90%以上が、説明なしでタスクの作成・完了・削除を成功させられる

## Assumptions

- ユーザー認証は不要であり、シングルユーザー向けのアプリとして動作する
- タスクデータはブラウザの IndexedDB を利用して永続化する（サーバーサイドのデータベースは不要）
- タスクのカテゴリ分け・優先度・期限日付けはv1のスコープ外とする
- モダンなウェブブラウザ（デスクトップ・モバイル両対応）での動作を前提とする
- 複数ユーザー間でのタスク共有はスコープ外とする
